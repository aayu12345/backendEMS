## Part 1: Detailed Component-by-Component User Flows

Whenever the interviewer asks: *"Show me the frontend screen. What happens when I do X? How does the data get here?"*, use these detailed flows.

### 1. `Login.jsx` (Authentication & Registration)
- **What it is:** The first screen the user sees. It contains forms for both "Sign In" and "Create Account".
- **User Action:** The user fills out their details to create an account and clicks Submit.
- **The React Side:** 
  - The `handleSubmit` function prevents the default page reload.
  - It checks `isLogin`. Because it's a registration, it builds a JSON payload: `{ name, email, password, role }`.
  - It sends an HTTP `POST` request to `http://localhost:8080/api/auth/register`.
- **The Backend Side:**
  - **Controller:** `AuthController.java` catches the `/api/auth/register` request using `@PostMapping("/register")`.
  - **Service:** It passes the payload to `AuthService.java`. The service generates a default `employeeId` if none exists, encrypts the password using `BCryptPasswordEncoder`, and saves it as a new `User` object.
  - **Database:** The `UserRepository` (which uses Hibernate/JPA) executes an `INSERT INTO users` SQL query into the NeonDB PostgreSQL database.
  - **Return:** A JWT (JSON Web Token) is generated by `JwtTokenProvider` and sent back to React.
- **The Result:** React receives the token, saves it to `localStorage` (so the user stays logged in even if they refresh), and loads the Dashboard.

### 2. `App.jsx` (The Main Router & Layout)
- **What it is:** The main wrapper component. It holds the Sidebar (for navigation) and manages the `activeTab` state. 
- **How it works:** 
  - On first load, `useEffect` checks if a token exists in `localStorage`. If yes, it sends a `GET` to `/api/auth/me` to verify the token is still valid.
  - It conditionally renders the other components based on the `activeTab` (e.g., if `activeTab === 'Employees'`, it renders `<EmployeeManager />`).

### 3. `Dashboard.jsx` (The Overview Screen)
- **What it is:** Shows summary statistics and a nice Chart.js bar graph.
- **User Action:** The user navigates to the Dashboard tab.
- **The React Side:** 
  - A `useEffect` hook triggers `fetchData()`, sending a `GET` request to `/api/reports/dashboard`.
  - It attaches the JWT Token to the request header (`Authorization: Bearer <token>`).
- **The Backend Side:**
  - **Controller:** `ReportController.java` catches `/api/reports/dashboard`.
  - **Service:** `ReportService.java` makes multiple database queries using different Repositories:
    - Queries `EmployeeRepository` to count total active employees.
    - Queries `AttendanceRepository` for today's date to count how many people have clocked in.
    - Queries `PayrollRepository` for the current month to sum up the total money paid.
  - **Return:** It combines all these stats into a single JSON object and returns it.
- **The Result:** React takes the numbers and updates the DOM (HTML), rendering the statistics cards and the Bar Chart.

### 4. `EmployeeManager.jsx` (Employee Directory)
- **What it is:** A data table showing all employees. Admins can add/edit/delete. Employees can only view.
- **Privacy Rule:** If an employee is logged in, their own salary is visible, but everyone else's salary is replaced with "Confidential".
- **User Action:** Admin clicks "Add Employee" and saves.
- **The React Side:** 
  - Sends a `POST` request to `/api/employees` with the form data.
- **The Backend Side:**
  - **Controller:** `EmployeeController.java`.
  - **Service:** `EmployeeService.java` receives the data. It auto-generates a unique `EMP-XXXX` ID. 
  - **Special Feature:** If the admin checked the "Generate Login Account" box, the service creates the `Employee` record **AND** a `User` record simultaneously, linking them by the `EMP` ID.
  - **Database:** `EmployeeRepository.save(employee)` runs the SQL `INSERT`.

### 5. `AttendanceTracker.jsx` (Clock In / Clock Out)
- **What it is:** Allows employees to record their daily work hours.
- **User Action:** An employee clicks "Clock In Session".
- **The React Side:** 
  - It calculates the current local time (e.g., `"09:15"`).
  - It sends a `POST` to `/api/attendance/mark` with `{ date, clockIn, status }`.
- **The Backend Side:**
  - **Security Filter:** Because the request doesn't contain the `employeeId`, the backend uses the `JwtAuthenticationFilter` to extract the user's email from the JWT token in the header.
  - **Service:** `AttendanceService.java` looks up the User by email to find their hidden `employeeId`. It checks if a record for today already exists. Since it's a clock-in, it creates a new `Attendance` record and saves it.
- **The Result:** The React component fetches the updated logs, and the "Clock In" button disables itself to prevent duplicate clicks.

### 6. `PayrollManager.jsx` (Payslip Generation)
- **What it is:** Manages monthly salaries, bonuses, and deductions.
- **User Action:** Admin clicks "Build List Logs" for June 2026.
- **The React Side:** Sends a `POST` to `/api/payroll/generate` with `{ month: "June", year: 2026 }`.
- **The Backend Side:**
  - **Service:** `PayrollService.java` pulls every active employee from the `EmployeeRepository`. 
  - It iterates over the list and dynamically creates a `Payroll` object for each person, setting their base salary, applying default bonuses/deductions (0), and setting the status to "Pending".
  - **Database:** `PayrollRepository.saveAll(list)` inserts them all into the database at once.

## Part 2: What Every Backend Annotation Does (`@`)

Spring Boot uses annotations (`@`) to avoid writing hundreds of lines of boilerplate setup code. Here is exactly what you should say when they point to one of these:

### The "Big Three" Setup Annotations
- **`@SpringBootApplication`**: Found in `Application.java`. It’s the "Start Button". It tells Spring to launch an embedded Tomcat web server and automatically scan your folders to wire up the rest of your application.
- **`@RestController`**: Put on Controller classes. It means two things: 1) "I am a controller that handles web requests", and 2) "Please take whatever Java object I return and automatically convert it into a JSON string before sending it to React."
- **`@Service`**: Put on Service classes. It tells Spring, "I hold the core business logic. Please create an instance of me when the app starts so the Controllers can use me."

### HTTP Routing Annotations (Controllers)
- **`@RequestMapping("/api/...")`**: Defines the base URL for an entire controller file.
- **`@GetMapping`, `@PostMapping`, `@PutMapping`, `@DeleteMapping`**: Maps specific HTTP methods to your Java functions. 
  - *Analogy:* `GET` is reading a menu. `POST` is ordering a new meal. `PUT` is changing your order. `DELETE` is canceling your order.
- **`@RequestBody`**: Placed inside the method parameters. It tells Spring to read the raw JSON coming from React and magically map it into a Java object (like your `LoginRequest` DTO).
- **`@PathVariable`**: Used when the URL itself contains data. E.g., for `/api/employees/{id}`, it extracts the `{id}` string from the URL bar so you can use it to search the database.

### Database Annotations (JPA / Hibernate)
These annotations map your Java code directly to PostgreSQL tables.
- **`@Entity`**: Tells the framework, "This Java class represents a Table in the database." Every new instance of this class will become a new row in the table.
- **`@Table(name = "users")`**: Overrides the default naming and forces the database table to be named "users".
- **`@Id`**: Marks a specific variable as the Primary Key (the unique identifier for the row).
- **`@GeneratedValue(strategy = GenerationType.IDENTITY)`**: Tells the PostgreSQL database to auto-increment the ID number (1, 2, 3...) whenever a new row is inserted.
- **`@Column(nullable = false, unique = true)`**: Configures the exact rules for the database column. "nullable = false" means it cannot be empty (NOT NULL in SQL), and "unique = true" means no two rows can share this value (prevents duplicate emails).
- **`@Enumerated(EnumType.STRING)`**: Used for Enums (like `Role.ADMIN` or `Role.EMPLOYEE`). It tells the database to store it as readable text ("ADMIN") instead of a confusing numeric integer (like `0` or `1`).

### Important Utility Annotations
- **`@Transactional`**: Used in Service methods that modify the database. It guarantees that if an error crashes the code halfway through saving multiple things, the entire database transaction is completely "rolled back" (undone). This prevents half-written, corrupted data.
- **`@Configuration` & `@Bean`**: Found in `SecurityConfig.java`. It tells Spring, "Run this method at startup, and save whatever object it returns into your internal memory." For example, we return a `BCryptPasswordEncoder` bean so the `AuthService` can use it to encrypt passwords.
- **`@JsonFormat(pattern = "yyyy-MM-dd")`**: Placed on LocalDate variables. It forces the Jackson library (which converts Java to JSON) to format the date exactly as `"2026-06-19"`. Without this, Java dates sometimes turn into confusing arrays like `[2026, 6, 19]` which breaks the React frontend logic.

### Interview Pro-Tips

1. **If they ask "Why React + Spring Boot?":**
   "We chose React because its component-based architecture allows us to build dynamic, fast, single-page UI components (like the Dashboard charts) without full page reloads. We chose Spring Boot because it provides incredibly robust security (Spring Security), easy database mapping (Spring Data JPA), and allows us to scale the backend cleanly using the Controller-Service-Repository pattern."

2. **If they ask "How do you handle security?":**
   "We use stateless JWT (JSON Web Tokens). When a user logs in, `AuthService` verifies their `BCrypt` hashed password. If correct, `JwtTokenProvider` creates a token. React stores this token in `localStorage` and sends it as a `Bearer` token in the `Authorization` header on every subsequent request. Our `JwtAuthenticationFilter` intercepts requests to verify the token before allowing access to the APIs."

3. **If they ask "How did you solve the clock in/out bug?":**
   "The frontend React logic relied on string comparison for the date (`todayString === log.date`). However, the backend was serializing the Java `LocalDate` object into a JSON array instead of a string. I debugged this by enforcing a strict string output using the `@JsonFormat(pattern = "yyyy-MM-dd")` annotation on the Entity, which immediately synced the frontend and backend states."
